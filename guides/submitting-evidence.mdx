# Submitting Evidence Guide

Learn how to properly submit evidence for AI-powered verification on AetherLock Protocol. This guide covers file preparation, upload processes, and optimization for successful verification.

## Understanding Evidence Requirements

### What is Evidence?

Evidence is the proof of work completion that you submit for AI verification. The AetherLock AI analyzes your submissions to determine if project requirements have been met according to the escrow terms.

### Evidence Types

<Tabs>
<Tab title="Visual Evidence">
**Screenshots**: UI/UX designs, website pages, app interfaces
**Images**: Logos, graphics, marketing materials, physical products
**Diagrams**: Architecture diagrams, flowcharts, wireframes
**Before/After**: Comparison images showing improvements

**Best Practices**:
- High resolution (minimum 1920x1080 for screenshots)
- Clear, well-lit images
- Multiple angles for physical products
- Annotated screenshots with callouts
</Tab>

<Tab title="Video Evidence">
**Screen Recordings**: Software demonstrations, website walkthroughs
**Live Demos**: Real-time functionality testing
**Presentations**: Project explanations and feature overviews
**Time-lapse**: Development process documentation

**Requirements**:
- Maximum 10 minutes per video
- 1080p resolution minimum
- Clear audio narration
- Stable recording (no shaky footage)
</Tab>

<Tab title="Document Evidence">
**Code Files**: Source code, configuration files, documentation
**Reports**: Analytics, performance metrics, test results
**Specifications**: Technical documentation, API docs
**Certificates**: Completion certificates, compliance reports

**Formats Supported**:
- PDF, DOC, DOCX, TXT, MD
- Code files: JS, TS, PY, RS, SOL, etc.
- Spreadsheets: XLS, XLSX, CSV
</Tab>

<Tab title="Interactive Evidence">
**Live URLs**: Deployed websites, web applications
**Repository Links**: GitHub, GitLab, Bitbucket repositories
**Demo Accounts**: Test credentials for applications
**API Endpoints**: Live API documentation and testing

**Security Notes**:
- Use temporary demo credentials
- Ensure public accessibility
- Include access instructions
- Set appropriate permissions
</Tab>
</Tabs>

<Tabs>

- **Videos**: 500 MB maximum per file
- **Documents**: 25 MB maximum per file
- **Archives**: 100 MB maximum per file

- Use efficient video codecs (H.264, H.265)
- Split large files into smaller segments
- Use cloud storage links for very large files

## IPFS Upload Process

### Understanding IPFS Storage

AetherLock uses IPFS (InterPlanetary File System) for decentralized evidence storage:

**Benefits**:
- Permanent, tamper-proof storage
- Decentralized availability
- Content-addressed (hash-based) retrieval
- Censorship resistance

**Process**:
1. Files uploaded to AetherLock interface
2. Automatically pinned to IPFS network
3. Hash generated for blockchain reference
4. Multiple IPFS nodes ensure availability

### IPFS Upload Steps

<Steps>
<Step title="Automatic IPFS Processing">
1. Upload files through AetherLock interface
2. System automatically processes files for IPFS
3. Generates content hashes for each file
4. Pins files to multiple IPFS nodes
5. Returns IPFS hashes for blockchain storage
</Step>

<Step title="Verification">
1. Receive IPFS hash for each uploaded file
2. Verify files are accessible via IPFS gateways
3. Check file integrity using content hashes
4. Confirm pinning status on multiple nodes
</Step>
</Steps>

### IPFS Hash Examples

```typescript

};

// Verify file accessibility
const verifyIPFSFile = async (hash: string) => {
  const gateways = [
    `https://ipfs.io/ipfs/${hash}`,
    `https://gateway.pinata.cloud/ipfs/${hash}`,
    `https://cloudflare-ipfs.com/ipfs/${hash}`
  ];
  
  for (const gateway of gateways) {
    try {
      const response = await fetch(gateway, { method: 'HEAD' });
      if (response.ok) return true;
    } catch (error) {
      continue;
    }
  }
  return false;
};
```

## Common Upload Issues and Solutions

### Upload Failures

<Tabs>
<Tab title="Network Issues">
**Problem**: Upload timeouts or connection errors

**Solutions**:
1. Check internet connection stability
2. Try uploading smaller batches of files
3. Use a wired connection instead of WiFi
4. Disable VPN temporarily
5. Try during off-peak hours

**Prevention**:
- Upload files in smaller batches
- Use stable internet connection
- Avoid uploading during network maintenance windows
</Tab>

<Tab title="File Size Issues">
**Problem**: Files exceed size limits

**Solutions**:
1. **Images**: Compress using tools like TinyPNG or ImageOptim
2. **Videos**: Use video compression software (HandBrake, FFmpeg)
3. **Documents**: Remove unnecessary images or split into multiple files
4. **Archives**: Split large archives into smaller parts

```bash
# Example: Compress video using FFmpeg
ffmpeg -i large-video.mov -vcodec libx264 -crf 28 compressed-video.mp4

# Example: Compress images in batch
for img in *.png; do
  pngquant --quality=65-80 --ext .png --force "$img"
done
```
</Tab>

<Tab title="Format Issues">
**Problem**: Unsupported file formats

**Solutions**:
1. Convert to supported formats:
   - PSD → PNG/JPG (Photoshop export)
   - AI → SVG/PNG (Illustrator export)
   - DOCX → PDF (universal compatibility)
   - Proprietary formats → Standard formats

**Conversion Tools**:
- Online converters (CloudConvert, Zamzar)
- Desktop software (GIMP, LibreOffice)
- Command-line tools (ImageMagick, Pandoc)
</Tab>
</Tabs>

### IPFS Issues

<Steps>
<Step title="Pinning Failures">
**Problem**: Files not properly pinned to IPFS

**Symptoms**:
- Upload completes but no IPFS hash generated
- Files not accessible via IPFS gateways
- Error messages about pinning services

**Solutions**:
1. Retry the upload process
2. Check file integrity (not corrupted)
3. Ensure file size is within limits
4. Contact support if issue persists
</Step>

<Step title="Gateway Access Issues">
**Problem**: Files uploaded but not accessible via IPFS gateways

**Troubleshooting**:
1. Try multiple IPFS gateways
2. Wait 5-10 minutes for propagation
3. Verify IPFS hash is correct
4. Check if file was properly pinned

**Alternative Gateways**:
- https://ipfs.io/ipfs/[hash]
- https://gateway.pinata.cloud/ipfs/[hash]
- https://cloudflare-ipfs.com/ipfs/[hash]
- https://dweb.link/ipfs/[hash]
</Step>
</Steps>

## Frequently Asked Questions

<AccordionGroup>
<Accordion title="How long do I have to submit evidence?">
The submission deadline is set when the escrow is created, typically 7-30 days after project completion. You'll receive email reminders 3 days and 1 day before the deadline. Late submissions may require client approval.
</Accordion>

<Accordion title="Can I update or add evidence after submission?">
Yes, you can add supplementary evidence until the verification process is complete. However, major changes or replacements may restart the AI verification process. It's best to submit complete evidence initially.
</Accordion>

<Accordion title="What happens if my evidence is rejected?">
If evidence is insufficient, you'll receive detailed feedback about what's missing or unclear. You can then submit additional evidence or clarifications. The client may also provide input on what they'd like to see.
</Accordion>

<Accordion title="Are there any restrictions on evidence content?">
Evidence must be directly related to the project requirements. Avoid including:
- Personal information of third parties
- Copyrighted material you don't own
- Inappropriate or offensive content
- Misleading or fraudulent information
</Accordion>

<Accordion title="How is my evidence stored and protected?">
Evidence is stored on IPFS for decentralization and permanence. Files are encrypted during transmission and access is controlled through the AetherLock platform. Evidence is retained for 1 year after escrow completion for dispute resolution purposes.
</Accordion>

<Accordion title="Can clients see my evidence before verification?">
Clients can see evidence summaries and descriptions but not the full files until after AI verification is complete. This prevents bias in the verification process while keeping clients informed of progress.
</Accordion>
</AccordionGroup>
